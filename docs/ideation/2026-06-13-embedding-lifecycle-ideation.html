Grounding Context

The finding that reframes everything

The headline worry — "the model isn't downloaded" — is mostly a development-build illusion. A fresh-context verifier confirmed that release builds already bundle the model inside the app: embedding-model.ts checks the packaged model directory first, so a shipped Interleave has a working model even though <userData>/models is empty. The verified reason a real install shows 0 of 25 embedded is that nothing ever starts indexing — pre-existing items never fire the per-item embed hook, the only reindex path is a manual "Build index" button, and startup just re-queues crashed jobs. On top of that, the "model ready" flag is set by a throwaway warm-up rather than an honest check, and embed-job failures are never shown anywhere.

So the gold-standard work is not "build a downloader." It is three things: (1) make indexing start and heal itself automatically, (2) make status honest and visible, and (3) prove the vectors are actually meaningful — because today a 100%-complete count can be 100% wrong.

What already exists (the ideas below build on it, they do not reinvent it):

  • A real background job system. A single Electron utilityProcess worker runs jobs at concurrency 2, with exponential backoff, crash recovery, and a persistent jobs table that survives restart. embed is already a registered job type.
  • An embedding service. enqueueElement() skips unchanged text via a content hash; reindexAll() can enqueue the whole corpus; results upsert transactionally into a sqlite-vec vector table. New extracts/cards/documents auto-enqueue an embed right after they're saved.
  • Solid storage. An embeddings bookkeeping table (with per-row model_id, content_hash, dim) plus a vec0 virtual table; nearest-neighbour search is filtered by model id; a real round-trip "is the vector engine actually working" smoke test guards against silent extension failures.
  • The model: EmbeddingGemma-300M (768-dim) run locally via transformers.js. Vendored into the app at build time; loaded offline at runtime. If it is ever absent, the code falls back to a deterministic hash-based embedder — which produces real-looking but meaningless vectors.

The verified gaps — what is missing today:

  • Nothing auto-starts indexing on launch, and there is no "download / acquire" path that isn't really just a warm-up. So a corpus that predates the feature stays at 0% forever.
  • No status surface in Settings. The only indicator is a "Build index (N of M)" hint on the Library screen that vanishes once you type a search.
  • Job failures are invisible — the error text is captured but never displayed, so indexing can be permanently stuck at "23 of 25" with no explanation.
  • No quality signal. Counts say "done"; nothing says "and the vectors are good." The hash fallback can fill the count to 100% while being semantically worthless.
  • No automatic re-embed when the model changes; the orphan/drift cleanup exists but is never scheduled; the runner ignores battery/idle state.

Topic Axes

The embedding lifecycle decomposed into five orthogonal surfaces. Each idea below targets one.

model-acquisition
Getting the model present & loadable: bundling vs download, integrity, offline, "ready" state.
indexing-pipeline
Turning content into vectors in the background: triggering, batching, throttling, prioritization.
index-health-migration
Keeping the index correct over time: idempotency, model-version changes, orphans, drift.
observability-surface
What the user can see & do: counts, ETA, model state, errors, actions — the Settings panel.
quality-eval
Proving the vectors are meaningful: smoke tests, drift checks — green/red confidence, not just counts.

1.

Self-Healing Index Supervisor

indexing-pipeline Confidence 92% Complexity Medium Recommended first

A small maintenance service in the Electron main process that owns the whole index "staying healthy" job. On launch (after the database, IPC, and job runner are up) it probes readiness, and if there are elements without an up-to-date embedding it quietly calls the existing reindexAll() — no button, no ceremony. It also runs a periodic low-priority scrub that prunes orphaned vectors, re-embeds items whose text changed, and re-embeds rows left on an old model. It registers in the backup restore/reset drain path so it never writes to a database that's about to be swapped out.

Axis
indexing-pipeline
Basis
direct: startup start() calls only recoverRunning() (re-queues crashed jobs), never a reindex sweep; reindexAll() exists but is manual-only. Pre-existing elements never fire the per-item auto-embed, so a corpus that predates the feature stays at 0%. pruneOrphanVectors() exists but is wired only to a manual cleanup, never scheduled. The "background maintenance lives in main + register in the backup drain path" pattern is already documented in docs/solutions/.
Rationale
This is the single change that converts "0 of 25, forever, silently" from the default state into a temporary state. Almost all the parts already exist; what's missing is the supervisor that starts and repeats them.
Downsides
Background CPU right after launch (mitigated by idea 7's throttle); must be carefully ordered against backup/restore and app shutdown so it never writes to a torn-down store.

2.

Honest Model-Ready Probe & Status Contract

model-acquisition Confidence 90% Complexity Low

Replace the "did a warm-up embed succeed?" heuristic with a real probe: on launch, check whether the bundled model directory exists, whether its .interleave-model-ready.json marker matches the active model id, and whether a one-shot embed actually returns the real model (not the hash fallback). Set the modelDownloaded state from that truth. Document the reality that release builds bundle the model, so the runtime "download" becomes a rare repair path (re-acquire only if the bundled copy is missing or corrupt) rather than the everyday mechanism the original premise assumed.

Axis
model-acquisition
Basis
direct: embeddingModelDownloaded is flipped only by a transient warm-up embed result, never by probing the marker file that the build step writes; the loader checks the packaged bundle dir first, so packaged builds already have a working model. external: Obsidian Smart Connections ships the model in the bundle (zero download) — the same posture this idea formalizes.
Rationale
The status surface (idea 3) is only as trustworthy as this flag. Fixing the probe is the cheapest change that makes "model: ready" actually mean ready, and it prevents over-investing in a downloader for a problem that mostly doesn't exist in shipped builds.
Downsides
A one-shot probe embed adds a little startup work (cache it, don't repeat every launch); the repair-download path still needs some minimal implementation for the rare corrupt-bundle case.
Confidence
90%
Complexity
Low
3.

"Search Intelligence" Health Panel in Settings

observability-surface Confidence 88% Complexity Medium

The centerpiece you asked for: a dedicated section in Settings, modeled on the existing AI Assistance panel (which already has a toggle, status chips, and a download button). It shows, in one always-accessible place: model state (bundled / ready / repairing, real model vs. fallback); indexed / total with a progress bar and an ETA ("~4 min remaining") derived from recent job throughput; a small readiness checklist (vector engine OK · model verified · dimensions match) instead of one opaque "enabled" boolean; a coverage-with-threshold reading that honestly labels results "partial coverage — keyword-weighted" below a reliability line; and actions (reindex, retry failed, prune, re-acquire model).

Axis
observability-surface
Basis
direct: Settings has no semantic panel today; the AI Assistance panel is a working analogue, and the UI primitives (status chips, progress bar, "OK" chip) already exist. The status payload already returns enabled / vecAvailable / modelDownloaded / embedded / total / modelId — most fields the panel needs exist; only ETA, the checklist breakdown, and coverage framing are new.
Rationale
This is the literal request ("see status, problems, what's downloaded, whether reindexing is happening"). Because the contract and primitives already exist, it is mostly a renderer addition, not new plumbing.
Downsides
ETA is noisy until a few jobs complete (show "estimating…" first); a status screen that updates live needs to subscribe to job events without re-querying on every tick.
Confidence
88%
Complexity
Medium
4.

Make Failures Visible & Recoverable

observability-surface Confidence 90% Complexity Low

Every embed job already carries an error field — it is simply never shown. Surface it: a failed-jobs count, the human-readable error text grouped by kind (model load failed, oversized item, worker crash, timeout), and a one-click "retry failed." Distinguish transient failures (retried automatically) from permanent ones (e.g. an item too large to embed) so the panel can say "3 items couldn't be indexed: [reason]" instead of stalling at "23 of 25" with no explanation.

Axis
observability-surface
Basis
direct: the job summary type has an error: string | null field that is populated by the runner but never rendered in the UI; after max retries a job is marked failed and effectively forgotten, with no surface.
Rationale
You explicitly asked the surface to show "problems." A permanently stuck count with no reason is exactly the confusing half-state to escape. The data already exists; only the routing to the UI is missing — which is why this is high-confidence, low-complexity.
Downsides
Needs a clear taxonomy of transient vs. permanent so "retry" isn't offered for things that will always fail; raw error strings must be translated to plain language for a non-technical reader.
Confidence
90%
Complexity
Low
5.

Prove It Works: Quality Canary + Fallback Quarantine

quality-eval Confidence 85% Complexity Medium

Two paired moves. The canary: after the model loads (or changes), embed a few known similar and dissimilar sentence pairs and assert similar pairs score high (≈≥0.75) and dissimilar pairs score low (≈≤0.30); show a green "embeddings verified" or red "degraded" badge. A cheap norm-distribution check over stored vectors catches corruption or mixed-model contamination. The quarantine: never present hash-fallback vectors as "related by meaning" (fall back to keyword search instead) and tag those rows provisional so they're re-embedded the moment the real model is ready.

Axis
quality-eval
Basis
direct: the fallback embedder is pure hash-bucketing — it returns valid-looking, unit-normalized 768-dim vectors with no semantic content, stored under a distinct fallback model id. The existing engine smoke test proves the vector store works, but nothing proves the model produces meaningful vectors. external: known-pair similarity smoke tests (≥0.75 / ≤0.30) and norm-distribution drift checks are standard lightweight embedding-quality signals.
Rationale
This is the subtle, high-value insight: a "fully indexed" badge can be a false green. For a knowledge tool, silently meaningless "related by meaning" results are the worst possible failure — they destroy trust in the core feature. The canary turns the status surface from a progress bar into a trust signal.
Downsides
The reference pairs are language/model specific (start with English, a handful of pairs); thresholds need tuning to avoid false "degraded" alarms.
Confidence
85%
Complexity
Medium
6.

Safe, Observable Model Upgrades (Re-Embed Campaign)

index-health-migration Confidence 84% Complexity Medium

When the embedding model id changes (a future better model, or escaping the hash fallback), automatically start a tracked, resumable re-embed campaign instead of waiting for someone to click a button. Nearest-neighbour search keeps serving the old generation's vectors until the new generation reaches a coverage threshold, then the system promotes the new generation and sweeps the old rows. Optionally, fold the model + text-pipeline version into the content hash so a change to how text is prepared also invalidates stale embeddings.

Axis
index-health-migration
Basis
direct: per-row model_id makes a model switch re-embeddable, and nearest-neighbour queries are already filtered by model id — but the only trigger is a manual reindex, so a model change leaves old vectors serving a silently shrinking subset until the user discovers the button. external: Postgres CREATE INDEX CONCURRENTLY — build the new index without blocking reads, swap at the end.
Rationale
This is what makes future model upgrades shippable at all. Without it, the day you ship a better model, users sit on a half-migrated, mixed-quality index with no signal — the same silent-corruption class the team already hit once with a migration.
Downsides
Briefly stores two generations of vectors (more disk during the campaign); the promote/sweep step must be transactional and crash-safe.
Confidence
84%
Complexity
Medium
7.

Scale-Ready Indexing: Power-Aware Throttle + Priority Order

indexing-pipeline Confidence 70% Complexity Medium Do later — when the corpus grows

Two improvements that are nearly irrelevant at 25 elements but become mandatory at thousands. Power/idle awareness: have the job runner watch Electron's power monitor — pause or slow background embedding on battery, burst when plugged in and idle — while foreground search-query embeds always run immediately. Priority ordering: have a full reindex embed the user's active working set first (recently opened, recently reviewed, due for review) so semantic search becomes useful for what they're touching now, with the cold archive backfilling behind it.

Axis
indexing-pipeline
Basis
direct: the runner uses a fixed concurrency with no power/idle awareness; the reindex query has no ordering (plain select up to a limit). external: Spotlight and Lightroom keep search usable while indexing and throttle on battery; search engines spend a finite "crawl budget" on high-value items first.
Rationale
You invited "super big / 1M-element" thinking. At that scale, a flat-out indexer is a hot, loud, battery-draining experience during reading, and FIFO ordering means the items you care about embed last. Designing the throttle/priority seam early is cheaper than retrofitting it. Honest caveat: at your current ~25 elements this is premature — sequence it after ideas 1–5.
Downsides
Adds a scheduling policy to tune (battery thresholds, idle windows); priority ordering couples the embedding pipeline to the attention scheduler, a cross-package dependency.
Confidence
70% (value certain; timing is "later")
Complexity
Medium
